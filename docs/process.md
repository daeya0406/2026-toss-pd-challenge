# 설계 프로세스 정리

책의 7단계 프레임워크에 「일정 맞추기」 프로젝트를 맞춰 정리했습니다.  
제출 문항(`제출문항.md`)과 연결해 보면 됩니다.

---

## 1. 왜 — 당신의 목표를 이해하라

### 목표 (한 문장)

> **6명 회의 조율에서, 조건의 영향력 차이를 반영해 주최자가 빠르게 시간을 정하고 참석자는 부담 없이 입력하게 한다.**

### 왜 이 문제인가

- 정보가 **없어서**가 아니라, **비교·결정 구조가 없어서** 조율이 반복됨
- 필수/선택, 불가/비선호가 **같은 무게**로 취급되면 주최자가 어디부터 봐야 할지 모름
- 참석자는 입력 부담이 크면 **미루거나 포기**함

### 성공하면 어떤 상태인가

- 주최자: 전원 가능 시간은 **바로 확정**, 나머지는 **누구에게 말할지** 알고 조율
- 참석자: 클릭만으로 일정 기록, 사유는 채팅으로
- 회의: 100% 합의 없이도 **필수 인원 기준으로 확정** 가능

## 2. 누가 — 오디언스를 정의하라

| 페르소나 | 역할 | 핵심 니즈 | 프로토타입 |
|----------|------|-----------|------------|
| **주최자 (김다연)** | 회의 만들기 · 확정 | 한 화면에서 6명 조건 비교·결정 | `create` → `merge` |
| **필수 참석자** (김민수, 이지영 등) | 일정 입력 | 최소 노력으로 “언제 안 되는지”만 전달 | `respond` |
| **선택 참석자** (박현우, 최수진, 정태호) | 일정 입력 | 불가해도 회의는 열릴 수 있음을 인지 | `respond` + merge 뱃지 |
| **미제출자** | (아직 입력 안 함) | 리마인드·나중에 입력 | merge 현황 · 일정 겹쳐보기 disabled |

### 1차 타겟

**주최자** — 병목이 “수집”이 아니라 **집단 조건 비교·결정**이기 때문.

### 2차 타겟

**참석자** — 입력률이 올라야 주최자 화면이 의미 있음 → 입력 마찰 최소화.

## 3. 언제 그리고 어디서 — 고객의 컨텍스트와 니즈를 이해하라

### 주최자 · 김다연

| When | Where | 니즈 | Pain |
|------|-------|------|------|
| 회의 전 주, 일정 요청 보낸 뒤 | 업무 중, 모바일 | 누가 제출했는지, 언제 확정할지 | 채팅·구두로 조건이 흩어짐 |
| 4명 제출 후 | 같은 주간 캘린더를 봐야 할 때 | “모두 되는 시간” vs “조율 필요” 구분 | 한 명씩 다시 물어보는 순환 |
| 후보가 애매할 때 | 회의 직전 | 특정 사람 일정만 깊게 보기 | 6명 데이터를 머릿속으로 합성하기 어려움 |

### 참석자 · 이지영, 박현우 등

| When | Where | 니즈 | Pain |
|------|-------|------|------|
| 링크 받은 직후 | 이동 중, 짧은 틈 | 빨리 끝내고 싶음 | 사유·장문 입력 요구 |
| 외근 있는 주 | 요일 단위로 먼저 생각 | “그날 전체 불가” 먼저 표시 | 시간 칸 하나씩만 막기 번거로움 |
| 이미 제출한 뒤 | 확정 전 | 수정 가능 | 덮어쓸지 확인 필요 |

### 컨텍스트에서 나온 설계 원칙

1. **넓은 단위 → 좁은 단위** (요일 불가 → 시간 그리드)
2. **언제만 받고, 왜는 조율로** (사유 필드 없음)
3. **평일 업무 시간** 기준 (주말 제외)
4. **링크만 공유**해도 이름으로 응답자 구분

## 5. 아이디어의 우선순위를 정하고 하나를 선택하라

### 평가 기준

- **차별성**: 챌린지에서 “왜 이 디자인인가”가 드러나는가
- **문제 적합성**: 주최자 병목(비교·결정)을 직접 푸는가
- **프로토타입 가능성**: 웹 HTML로 설득 가능한가

### 우선순위

| 순위 | 아이디어 | 선택 |
|------|----------|------|
| 1 | A + B **일정 맞추기 (히트맵 + 겹쳐보기)** | ✅ **핵심 솔루션** |
| 2 | C 필수/선택 | ✅ 규칙으로 포함 |
| 3 | D + E 참석자 입력 구조 | ✅ 플로우로 포함 |
| 4 | G 조율 메시지 | ✅ merge 4단계로 스케치 |
| 5 | F 이름 게이트 | ✅ 구현 |
| 6 | I 자유 서술 | ❌ 의도적 제외 (조율은 채팅) |
| 7 | J API 연동 | ❌ 범위 밖 (프로토타입) |

### 선택한 하나 (대표 아이디어)

> **「6명 일정을 같은 타임테이블에 겹쳐 보기」**  
> 히트맵으로 빠르게 결정하고, 레이어 뷰로 검증한다.

나머지(C, D, E, F, G)는 이 핵심을 **지원하는 서브 솔루션**.

---

## 6. 어떻게 — 솔루션을 만들어라

### 스토리보드 (Happy path)

```
[주최자] create: 회의 이름 → 기간 → 참석자(필수/선택) → 요청 발송
    ↓ 링크
[참석자] respond: 이름 확인 → 요일 불가 → 시간 그리드 → 완료
    ↓ 집계
[주최자] merge-1: 입력 현황 (4/6 완료)
    ↓
[주최자] merge-2: 히트맵 · 수 14:00 「전원 가능」 선택 → 확정
    ↓ (또는)
[주최자] merge-2: 애매한 칸 → 「개별 일정 겹쳐보기」
[주최자] merge-6: 이름 토글 · 레이어 확인 → 히트맵으로 복귀 → 확정
    ↓
[주최자] merge-5: 확정 완료 (참석/조율중/불참 표시)
```

### 태스크 정의

**주최자 — 회의 만들기**

1. 회의 이름 입력
2. 조율 기간 선택 (평일, 최대 2주)
3. 참석자 추가 · 필수/선택 지정
4. 검토 후 일정 입력 요청 발송

**참석자 — 일정 입력**

1. 이름 확인 (목록 매칭)
2. (선택) 요일 전체 불가 설정
3. 시간 그리드에 불가/비선호 드래그
4. 완료 확인 (바텀시트)

**주최자 — 가능한 시간**

1. 제출 현황 확인
2. 히트맵에서 후보 슬롯 선택
3. (선택) 일정 겹쳐보기로 레이어 검증
4. 확정 또는 1명 조율 요청
5. 결과 확인

### 스케치 → 프로토타입 매핑

| 스케치 의도 | 파일 |
|-------------|------|
| 플로우 맵 · 리뷰 가이드 | `index.html` |
| UI 스펙 · 상태 | `components.html` |
| 회의 만들기 | `create.html` |
| 일정 입력 | `respond.html` |
| 히트맵 · 일정 겹쳐보기 · 확정 | `merge.html` |
| 토스트 · 바텀시트 | `overlays/view.html` |

## 7. 어떻게 — 성공을 측정하라

### 제품 지표 (가설)

| 지표 | 정의 | 좋아지면 |
|------|------|----------|
| **입력 완료율** | 초대 대비 제출 비율 | 참석자 마찰 감소 |
| **확정 소요 시간** | 요청 발송 → 확정까지 | 주최자 조율 반복 감소 |
| **첫 후보 선택률** | 히트맵 「전원 가능」 칸에서 바로 확정 | 비교·결정 구조가 작동 |
| **일정 겹쳐보기 사용률** | merge-2 → merge-6 진입 비율 | 애매한 케이스에서 검증 도구로 쓰임 |
| **조율 메시지 발송률** | merge-4 진입 후 요청 비율 | 1~2명 이슈 시 타겟 조율 |

### UX 품질 지표 (프로토타입·테스트용)

| 지표 | 측정 방법 |
|------|-----------|
| **3분 내 플로우 이해** | `index.html` 리뷰 가이드로 심사자/동료 테스트 |
| **「누구에게 말해야 하는지」 인지** | 히트맵 이름 뱃지 보고 대상 말하기 (5초 내) |
| **입력 부담 인지** | 참석자 플로우 후 “얼마나 걸렸는지” 주관 평가 |
| **확정 신뢰도** | 필수/선택 규칙 이해 후 확정 의사 (인터뷰) |

### 이번 프로토타입에서 검증한 것 / 안 한 것

| 검증함 ✅ | 아직 안 함 (다음 단계) |
|-----------|------------------------|
| 히트맵 + 일정 겹쳐보기 IA | 실사용자 A/B |
| 필수/선택 규칙 UI | 입력 완료율 수치 |
| 참석자 최소 입력 플로우 | 조율 후 실제 참석률 |
| 대표 validation · 피드백 | 로딩 · 에러 전체 구현 |
