# 문제 정의

6명이 다음 주 안에 1시간 회의를 잡을 때 발생하는 조율 문제를, **역할별 상황**으로 정리했습니다.

---

## 상황 A · 주최자 — 정보가 흩어져 있다

**누가:** 회의를 만드는 주최자  
**언제:** 6명 일정을 받은 뒤, 가능한 시간을 고를 때  
**무엇이:** 불가·비선호·필수/선택 여부가 채팅·구두·개인 캘린더에 흩어져 있음  
**결과:** “모두 되는 시간”을 찾다가 한 명씩 다시 확인하는 **순환 조율**이 반복됨

> 병목은 “일정을 아직 안 받았다”가 아니라, **받은 조건을 한 화면에서 비교·결정하지 못하는 것**입니다.

---

## 상황 B · 참석자 — 입력 부담이 크다

**누가:** 링크를 받은 참석자  
**언제:** 바쁜 업무 중 일정 입력 요청을 받았을 때  
**무엇이:** 슬롯마다 사유까지 적으라 하면 부담 → 미루거나 포기  
**결과:** 실제로는 “언제 안 되는지”만 알려주고, 맥락은 채팅으로 따로 전달함

> 사람은 **넓은 단위(외근 요일)** 부터 생각하고, 세부 시간은 그다음에 떠올립니다.

---

## 상황 C · 조건의 영향력이 다른데 한 줄로 취급된다

| 조건 | 회의 성립에 미치는 영향 |
|------|------------------------|
| 필수 3명 모두 가능 | 바로 확정 후보 |
| 필수 1명 불가 | 해당 시간 **후보 제외** |
| 선택 1명 불가 | 조율 대상 (회의는 열 수 있음) |
| 필수 비선호 | 조율 우선순위 ↑ |
| 선택 비선호 | 참고 후 조율 |

**기존 방식의 한계:** 위 조건이 동일한 무게로 취급되어, 주최자가 **어디를 먼저 봐야 하는지** 알기 어렵습니다.

---

## 핵심 문제 (한 문장)

> **조건의 영향력이 다른데 우선순위 없이 한 줄로 취급되어**, 주최자는 순차 조율을 반복하고, 참석자는 입력을 미룹니다.

---

## 해결 방향 (요약)

| 역할 | 해법 |
|------|------|
| 참석자 | 구조화 입력만 (불가/비선호) · 사유는 조율 단계로 |
| 주최자 | 6명 결과를 **한 타임테이블에 겹쳐 보기** |
| 조율 | 필수/선택 규칙 + 이름 뱃지로 **누구에게 말할지** 표시 |

자세한 설계 결정은 [`decisions.md`](./decisions.md), 화면별 상태는 [`states.md`](./states.md), 7단계 프로세스 정리는 [`process.md`](./process.md)를 참고하세요.
