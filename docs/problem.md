# 문제 정의

6명이 다음 주 안에 1시간 회의를 잡을 때, **각자 일정은 있는데 한 타임테이블에 겹쳐 보이지 않는** 문제를 정리했습니다.

---

## 핵심 문제 (한 문장)

> **6명의 일정을 같은 주간 그리드 위에 겹쳐 비교할 수 없어**, 주최자는 머릿속으로 합치거나 한 명씩 다시 물어봐야 한다.

---

## 왜 문제인가

| 기존 방식 | 한계 |
|-----------|------|
| 개인 캘린더 | 6명 일정을 **한 화면에 겹쳐** 보기 어려움 |
| 일정 투표·표 | “누가 언제 가능한지”만 모음 · **같은 칸에 어떻게 겹치는지** 안 보임 |
| 채팅·구두 조율 | 조건이 말로만 전달됨 · **공간(그리드)으로 합성되지 않음** |

그 결과 주최자는 후보를 고르다 한 명씩 “이 시간 되세요?”를 반복하거나, 머릿속으로 6명 일정을 합쳐 봐야 합니다.

---

## 상황 A · 주최자 — 겹쳐 볼 수 없다

**누가:** 회의를 만드는 주최자  
**언제:** 6명 일정을 받은 뒤, 가능한 1시간을 고를 때  
**무엇이:** 6명의 불가·비선호·필수/선택 정보가 **한 주간 타임테이블 위에 겹쳐지지 않음**  
**결과:** 비교 없이 순차 확인 · 머릿속 합성

> 과제가 요구하는 건 “표로 모으기”가 아니라, **동료들의 상황을 한눈에 파악**하는 경험입니다.

---

## 상황 B · 같은 칸이라도 우선순위가 다르다

6명을 한 그리드에 올리면, **같은 시간 칸**이라도 조건 조합에 따라 회의에 미치는 영향이 다릅니다.

| 조건 | 회의에 미치는 영향 (우선순위) |
|------|------------------------------|
| 전원 가능 | 바로 고를 수 있음 |
| 필수 1명 불가 | 해당 시간 **막힘** (후보 제외) |
| 선택 1명 불가 | **조율**하면 회의 가능 |
| 필수 비선호 | 비선호 중 **먼저** 말할 대상 |
| 선택 비선호 | 참고 후 조율 |

기존 방식은 이 차이를 **같은 무게로 나열**해서, 겹쳐 봐도 **어떤 조건을 먼저 봐야 하는지** 순서가 안 보입니다.

---

## 관련 맥락 · 참석자 입력 (문제 정의 보조)

겹쳐 보려면 6명 입력이 먼저 모여야 합니다. 참석자는 슬롯마다 길게 쓰라 하면 미루고, 실제로는 “언제 안 되는지”만 알립니다.  
이건 **별도 문제**이지만, 입력을 짧게 받지 않으면 주최자가 겹쳐 볼 데이터 자체가 비어 있습니다. → 해결 설계는 [`decisions.md`](./decisions.md) 참고.

---

## 해결 방향 (요약)

| 단계 | 해법 |
|------|------|
| 입력 | 6명이 **동시에** 구조화 입력 (불가/비선호만 · 요일 → 시간) |
| 집계 | 같은 주간 타임테이블에 **겹쳐 보기** (히트맵) |
| 검증 | 애매한 칸은 **개별 일정 겹쳐보기** (레이어 토글) |
| 읽기 규칙 | 필수/선택·불가/비선호를 색·뱃지로 구분 |

경우의 수 상세: [`heatmap-rules.md`](./heatmap-rules.md)
