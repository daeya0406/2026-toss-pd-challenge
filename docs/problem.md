# 문제 정의

처음 아이디어는 단순했다. **6명 일정을 같은 주간 그리드에 겹쳐 보면 된다.**

6명이 다음 주 안에 1시간 회의를 잡을 때, 각자 일정은 있는데 **한 타임테이블에 겹쳐 보이지 않는** 문제부터 정리했습니다.

---

## 핵심 문제 (한 문장)

> **6명의 일정을 같은 주간 그리드 위에 겹쳐 비교할 수 없어**, 주최자는 머릿속으로 합치거나 한 명씩 다시 물어봐야 한다.

---

## 왜 문제인가

| 기존 방식 | 한계 |
|-----------|------|
| 개인 캘린더 | 6명 일정을 **한 화면에 겹쳐** 보기 어려움 |
| 일정 투표·표 | “누가 언제 가능한지”만 모음 · **같은 칸에 어떻게 겹치는지** 안 보임 |
| 채팅·구두 조율 | 조건이 말로만 전달됨 · **그리드로 합성되지 않음** |

그 결과 주최자는 후보를 고르다 한 명씩 “이 시간 되세요?”를 반복하거나, 머릿속으로 6명 일정을 합쳐 봐야 합니다.

---

## 상황 A · 주최자 — 겹쳐 볼 수 없다

**누가:** 회의를 만드는 주최자  
**언제:** 6명 일정을 받은 뒤, 가능한 1시간을 고를 때  
**무엇이:** 6명의 불가·비선호·필수/선택 정보가 **한 주간 타임테이블 위에 겹쳐지지 않음**  
**결과:** 비교 없이 순차 확인 · 머릿속 합성

---

## 설계에서 힘들었던 것 · 겹치면서 나오는 경우의 수

겹치기만 하면 끝이 아니라, **칸마다 어떻게 보여줄지** 규칙을 하나씩 정해야 했다.

| 겹치면서 생기는 조합 | 화면에서 정한 것 (예) |
|---------------------|----------------------|
| 전원 가능 | 파랑 · 「전원 가능」 |
| 필수 1명 불가 | 회색 · 후보 제외 |
| 선택 1명 불가 | 이름 뱃지 · 조율 대상 |
| 필수/선택 비선호 | 노랑 · 필수는 `*` |
| 요일 전체 불가 vs 시간 칸 불가 | 필수는 열 회색 / 선택은 그 사람만 |
| 불가+비선호 혼재, 비선호 3명+ | **가능한 시간**에서는 비우고 **일정 겹쳐보기**로 |
| 색이 섞이면 읽기 어려움 | 불가 → 비선호 → 가능 순 (섞지 않음) |

이 표는 “처음부터 풀려고 잡은 추상 문제”가 아니라, **겹쳐 보는 화면을 만들다가** 맞닥뜨린 세부 규칙이다.

자세한 경우의 수: [`heatmap-rules.md`](./heatmap-rules.md)

---

## 관련 맥락 · 참석자 입력

겹쳐 보려면 6명 입력이 먼저 모여야 합니다. 참석자는 슬롯마다 길게 쓰라 하면 미루고, 실제로는 “언제 안 되는지”만 알립니다.  
입력을 짧게 받지 않으면 주최자가 겹쳐 볼 데이터 자체가 비어 있습니다. → [`decisions.md`](./decisions.md) 참고.

---

## 해결 방향 (요약)

| 단계 | 화면 | 해법 |
|------|------|------|
| 입력 | 일정 입력 | 6명이 동시에 불가/비선호만 · 요일 → 시간 |
| 집계 | **가능한 시간** (참여율 높은 시간) | 같은 주간 그리드에 겹쳐 보기 |
| 검증 | **일정 겹쳐보기** | 참여자 이름 토글 · 레이어 확인 |
| 읽기 규칙 | 위 두 화면 | 필수/선택·색·뱃지 ([`heatmap-rules.md`](./heatmap-rules.md)) |

설계 결정: [`decisions.md`](./decisions.md) · 화면 상태: [`states.md`](./states.md) · 프로세스: [`process.md`](./process.md)
