# 설계 결정 기록

「일정 맞추기」 프로토타입의 주요 UX 결정과 **이유**입니다.

> **출발점:** 6명 일정을 같은 주간 그리드에 **겹쳐 보기**.  
> 아래 결정들은 겹친 화면을 만들면서, **칸마다 어떻게 읽을지** 맞닥뜨린 경우의 수를 정리한 것입니다.

---

## 1. 역할 분리: 주최자 vs 참석자

**결정:** `create` + `merge`(주최자) / `respond`(참석자) 플로우 분리  
**이유:** 참석자는 최소 입력만, 주최자는 비교·결정에 집중. 같은 앱 안에서 역할이 섞이면 인지 부담이 커짐.

---

## 2. 참석자 입력 — 「언제」만, 「왜」는 나중에

**결정:** 불가(빨강) / 비선호(노랑)만 받고 사유 필드 없음  
**이유:** 사유까지 구조화하면 입력 포기율이 올라감. 맥락은 조율 단계에서 채팅 1~2회로 충분하다고 가정.

---

## 3. 요일 전체 불가 → 시간 그리드 순서

**결정:** respond에서 요일 단위 차단 후 시간 그리드  
**이유:** 외근 등은 “그날 전체”로 먼저 떠오름. 넓은 단위 → 좁은 단위가 실제 사고 순서와 맞음.

---

## 4. 필수 / 선택 참석

**결정:** create에서 참석자마다 필수·선택 지정, 히트맵·확정 규칙에 반영  
**이유:** 100% 합의 없이도 회의를 열 수 있어야 함. 선택 인원 불가는 조율 대상, 필수 불가는 후보 제외.

---

## 5. 히트맵 — 요일 전체 불가 vs 슬롯

**요일 전체 불가 (외근):**

| 역할 | 히트맵 |
|------|--------|
| 필수 | 해당 요일 **열 전체 회색** |
| 선택 | 열 유지 · 해당 인원만 전 시간 `unavail` → 이름 뱃지 |

**이유:** 선택 인원은 불참해도 회의를 잡을 수 있어야 함. 필수만 열 단위로 막는다.

자세한 경우의 수: [`heatmap-rules.md`](./heatmap-rules.md)

---

## 6. 히트맵 — 이름 뱃지 규칙

**결정:**

| 조건 | 표시 |
|------|------|
| 전원 가능 | 「전원 가능」 |
| 필수 불가 / 3명 미만 ok | 회색 빈 칸 |
| 선택만 불가 | 이름 1명 |
| 비선호만 | 이름 최대 2명 |
| 불가+비선호 혼재 / 비선호 3명+ | 빈 칸 |

**이유:** 히트맵은 **빠른 결정**용. 복잡한 칸은 일정 겹쳐보기로 넘김. 「불가」 텍스트는 시각 노이즈라 제거.

---

## 7. 개별 일정 겹쳐보기

**결정:** merge 2단계 하단 CTA로 진입 · 이름 토글 · 레이어 합성 · 확정은 히트맵에서만  
**이유:**

- 히트맵 = 요약·결정
- 일정 겹쳐보기 = 탐색·검증
- 미제출자는 토글 비활성 (데이터 없음)
- 한 칸에 색 하나: 불가 > 비선호 > 가능, 인원당 +20% 농도

---

## 8. 참석자 행 UI (create 3단계)

**결정:** `[필수/선택 select]` → `[이름]` → `[삭제]`  
**이유:** 역할 배지가 먼저 오면 열이 맞고, 이름 길이와 무관하게 스캔하기 쉬움. 주최자 행은 삭제 불가.

---

## 9. 검증 · 피드백 톤

**결정:**

- 치명적이지 않은 누락 → **warning 토스트** (이름 미입력 등)
- 되돌리기 어려운 액션 → **바텀시트 확인**
- 필드 단위 → **인라인 에러** (respond 이름)

**이유:** 토스 패턴에 맞게 가볍게 막되, 확정·발송은 한 번 더 확인.

---

## 10. 의도적으로 하지 않은 것

| 항목 | 이유 |
|------|------|
| 사유 재구조화 | 조율은 사람이 한다 |
| 100% 합의 대기 | 필수/선택으로 확정 가능 |
| 주말 달력 표시 | 회의 후보는 평일 업무 시간 기준 |
| 전 필드 validation 구현 | 프로토타입 — 대표 케이스만 구현, 나머지는 `states.md` |
